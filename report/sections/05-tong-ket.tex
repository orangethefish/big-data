\section{Summary}

\subsection{Discussion}

This project implemented a complete text-first pipeline for harmful video detection using weak supervision and pseudo-labeling. The main contribution is not only the final classifier, but the full engineering workflow around it: heterogeneous data ingestion, source-specific cleanup, canonical text consolidation, auditability, leakage-safe splitting, agreement analysis, model comparison, and local inference through a demo API.

\subsubsection{What Worked Well}

The data engineering design worked well for the messy structure of the dataset. The bronze, silver, and gold layer pattern made the pipeline easier to reason about because each stage had a clear purpose. Raw data preservation, schema normalization, and modeling-ready consolidation were separated instead of mixed together. This separation is important for harmful-content projects because small data-cleaning decisions can directly change model behavior.

The canonical wide table also became a useful modeling contract. By keeping one row per video identifier and selecting the longest non-empty title, description, and transcript fields, the pipeline preserved richer text when the same video appeared in multiple sources. At the same time, duplicate and merge auditing made this process traceable.

The agreement analysis was also valuable. The measured Cohen's kappa values show that GPT-4-Turbo and crowdworker labels do not behave like clean ground truth. This supports the weak-supervision design: the weak sources are useful, but they must be weighted and filtered rather than blindly merged into the training set.

From a modeling perspective, weak supervision provided a modest but consistent improvement over the expert-only baseline. On the expert test split, macro-F1 improved from 0.5701 to 0.5740, and harmful-class recall improved from 0.9712 to 0.9726. The improvement is small, but it is meaningful because the weak labels were accepted conservatively and the validation/test split was protected from leakage.

\subsubsection{What Did Not Work As Well}

Pseudo-labeling did not improve the final model. Although the unlabeled pool contained many text-bearing rows and the pseudo-labeling step accepted 9254 high-confidence examples, the accepted labels were extremely imbalanced: 9253 were harmful and only 1 was harmless. After retraining, the pseudo-label model reduced validation macro-F1 compared with the weak-supervision model. Under the project's fallback rule, the pseudo-label model was therefore not selected.

This negative result is still useful. It shows that increasing the amount of training data is not automatically beneficial when the added labels are generated by the model itself. Pseudo-labeling can amplify a model's existing bias, especially when the accepted examples are concentrated in one class. In this project, the validation-based selection rule prevented that weaker model from becoming the shipped model.

The final model also has an important precision-recall tradeoff. It achieves very high harmful-class recall, but it still produces many false positives at the default threshold. This is acceptable for a triage-oriented prototype where missing harmful content is costly, but it would require careful threshold tuning before use in a real moderation workflow. A review team with limited capacity might prefer a higher threshold to reduce false positives.

\subsubsection{Limitations}

The first limitation is dataset scope. The system is designed for streaming-platform-style harmful video detection, but the experiments are evaluated only on YouTube-derived MetaHarm data. Therefore, the report should not claim that the trained model is proven to generalize across all streaming platforms. The stronger claim is that the pipeline design is generalizable and can be adapted to other platforms if comparable text and label sources are available.

The second limitation is modality. The current model uses only title, description, and transcript. This makes the pipeline reproducible and easier to run locally, but it also means that the model cannot understand visual content, on-screen text, audio tone, background imagery, or video-frame context. Some harmful videos may be visually harmful even when their title and description appear harmless.

The third limitation is label ambiguity. Harmful-content annotation is subjective and policy-dependent. The low agreement values between label sources show that even human and machine annotators can disagree substantially. Some false positives and false negatives may therefore reflect genuine ambiguity rather than simple model error.

The fourth limitation is that the current deliverable is binary classification only. The system can identify whether a video is likely harmful, but it does not yet explain which harm category is present. Category-level prediction would be useful for moderation workflows, but it requires an additional phase of modeling and evaluation.

\subsubsection{Ethical and Practical Considerations}

Harmful-content detection can affect visibility, review priority, and moderation outcomes. A false positive may unfairly flag harmless content, while a false negative may allow harmful content to remain undetected. Because of this, the model should be interpreted as a decision-support tool rather than an automatic enforcement mechanism.

The system should also be used with awareness of dataset bias. A model trained on one platform's data may learn platform-specific language, annotation patterns, or cultural assumptions. If the system is adapted to another platform, it should be revalidated on data from that platform before any operational use.

Finally, the prediction probability should not be treated as a complete explanation. It is a useful ranking signal, but reviewers may still need examples, highlighted text, category predictions, or other explanations to understand why a video was flagged.

\subsection{Conclusion}

The project successfully built a reproducible, text-first harmful video detection pipeline with weak supervision, pseudo-labeling, model comparison, reporting, and a local prediction API. The final selected model is \texttt{expert\_plus\_weak}, because weak supervision improved validation and test behavior while pseudo-labeling reduced validation macro-F1.

The results show that weak supervision can provide a small improvement over an expert-only supervised baseline when weak labels are filtered carefully and leakage is controlled. The pseudo-labeling experiment demonstrates a different lesson: high-confidence unlabeled examples must still be validated because they can introduce class imbalance and reduce generalization.

Overall, the completed system meets the main goals of the project. It produces reproducible processed datasets, documents data quality issues, compares multiple model variants fairly, prioritizes macro-F1 and harmful-class recall, reports pseudo-labeling honestly as a negative result, and exposes the selected model through a simple local API.

\subsubsection{Future Work}

Several extensions can improve the system:

\begin{itemize}
    \item Implement six one-vs-rest harm-category classifiers for \texttt{IH}, \texttt{HH}, \texttt{CB}, \texttt{ADD}, \texttt{SXL}, and \texttt{PH}.
    \item Add threshold tuning based on practical moderation costs and review capacity.
    \item Improve probability calibration and pseudo-label selection to reduce class imbalance.
    \item Explore stronger text models, such as transformer-based encoders, if compute resources allow.
    \item Add multimodal signals such as thumbnails, sampled frames, or audio-derived features after the text pipeline is stable.
    \item Evaluate the pipeline on non-YouTube or cross-platform datasets to test generalization more directly.
    \item Add explainability features that highlight influential title, description, or transcript terms for human reviewers.
    \item Improve container runtime logging and verify execution on additional hardware architectures.
\end{itemize}

These future directions would move the project from a strong local batch prototype toward a more complete research platform for harmful-video detection.
