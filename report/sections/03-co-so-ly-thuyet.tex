\section{Theoretical Background}

\subsection{Harmful Video Detection}

Harmful video detection is a content classification problem in which a system estimates whether a video contains or promotes unsafe, abusive, misleading, exploitative, or otherwise policy-violating material. In a practical moderation setting, this kind of model is usually not expected to make final enforcement decisions by itself. Instead, it can support triage by identifying content that should receive closer review.

This project formulates the core task as binary classification:

\begin{equation}
    y =
    \begin{cases}
        1, & \text{if the video is harmful},\\
        0, & \text{if the video is harmless}.
    \end{cases}
\end{equation}

The binary formulation is chosen as the main deliverable because it is simpler, easier to evaluate reliably, and more suitable for building a stable end-to-end data pipeline. A more detailed multi-label formulation is possible when videos need to be assigned to specific harm categories. In this project, the six category labels \texttt{IH}, \texttt{HH}, \texttt{CB}, \texttt{ADD}, \texttt{SXL}, and \texttt{PH} are treated as a phase-two extension rather than part of the completed core system.

\subsection{Text-First Feature Design}

Videos contain multiple possible information sources, including visual frames, audio, metadata, comments, engagement signals, and text descriptions. This project intentionally starts with a text-first design. The model input is limited to three fields:

\begin{itemize}
    \item \textbf{Title}: a short summary written for viewers.
    \item \textbf{Description}: longer contextual text provided with the video.
    \item \textbf{Transcript}: spoken content converted into text when available.
\end{itemize}

The combined model text can be represented as:

\begin{verbatim}
model_text = normalize(title)
           + normalize(description)
           + normalize(transcript)
\end{verbatim}

Empty fields are skipped, and whitespace is normalized before modeling. This design keeps the system reproducible and reduces dependence on external tools for image extraction, audio processing, or platform-specific scraping. It also reduces leakage risk by excluding features such as channel identity, view count, and duration. Channel identity can cause the model to learn creator-level patterns instead of content-level signals, while popularity and duration fields are not consistently available across all labeled sources.

\subsection{Bronze, Silver, and Gold Data Layers}

Reliable machine learning depends on reliable data engineering. This project uses a three-layer data organization pattern: bronze, silver, and gold. Each layer has a different role in the pipeline.

The bronze layer supports traceability because it preserves what was ingested. The silver layer supports consistency because every source is converted into the same schema. The gold layer supports modeling because it produces one canonical record per video identifier and prepares clean inputs for training and evaluation.

This layered design is especially useful when source files are heterogeneous. In this project, the data includes spreadsheets, CSV files, different annotator sources, duplicate files, missing video identifiers, inconsistent column names, and variable text availability. A layered pipeline makes these issues visible instead of hiding them inside model training code.

\subsection{Weak Supervision}

Weak supervision is a learning strategy that uses imperfect label sources to increase the amount of training data. These labels may be cheaper, noisier, or less authoritative than expert labels. In this project, domain expert labels are treated as the primary ground truth, while GPT-4-Turbo and crowdworker labels are treated as weak sources.

The key idea is not to trust every weak label equally. Instead, the system estimates each weak source's reliability by comparing it with domain expert labels on the expert-labeled training overlap:

\begin{equation}
    r_s = \frac{\#\text{ agreements between source }s\text{ and domain expert}}{\#\text{ overlapping training examples}}.
\end{equation}

The reliabilities are then normalized into weights:

\begin{equation}
    w_{\text{GPT}} =
    \frac{r_{\text{GPT}}}{r_{\text{GPT}} + r_{\text{Crowd}}},
    \qquad
    w_{\text{Crowd}} =
    \frac{r_{\text{Crowd}}}{r_{\text{GPT}} + r_{\text{Crowd}}}.
\end{equation}

For a video with available weak labels, the weak harmful probability is a weighted average:

\begin{equation}
    p_{\text{weak}} =
    \frac{\sum_{s \in S} w_s y_s}{\sum_{s \in S} w_s},
\end{equation}

where \(S\) is the set of available weak sources and \(y_s \in \{0,1\}\) is the binary label from source \(s\). The system then applies a conservative acceptance rule:

\begin{verbatim}
if weak_p >= 0.75:
    accept as harmful
elif weak_p <= 0.25:
    accept as harmless
else:
    abstain
\end{verbatim}

Accepted weak labels are given sample weights based on confidence:

\begin{equation}
    \text{sample weight} = 0.5 + |p_{\text{weak}} - 0.5|.
\end{equation}

This gives higher weight to weak labels farther from uncertainty and lower weight to labels closer to the decision boundary.

\subsection{Pseudo-Labeling}

Pseudo-labeling is a self-training method in which a trained model is used to assign labels to previously unlabeled examples. These automatically generated labels are then added to the training set if they satisfy confidence requirements. The purpose is to use the unlabeled pool without manually annotating every example.

In this project, pseudo-labeling is deliberately conservative. The model first scores unlabeled videos using the best weak-supervision model. A pseudo-label is accepted only if the example has enough text and the calibrated harmful probability is very close to either 0 or 1:

\begin{verbatim}
if token_count >= 30 and (p_harmful >= 0.95 or p_harmful <= 0.05):
    accept pseudo-label
else:
    discard
\end{verbatim}

All accepted pseudo-labels receive a fixed sample weight of 0.5. The model is retrained once, and the self-training process stops after that single round. This prevents uncontrolled feedback loops where model errors can be repeatedly reinforced.

The final pseudo-labeling rule is validation-based. The pseudo-label model is kept only if it does not reduce validation macro-F1 or harmful-class recall compared with the weak-supervision model. This rule is important because pseudo-labeling can increase training size while still reducing model quality if the accepted labels are biased or incorrect.

\subsection{TF-IDF Text Representation}

Machine learning models require numeric input, so text must be converted into feature vectors. This project uses term frequency-inverse document frequency, commonly called TF-IDF. TF-IDF gives higher weight to terms that are frequent in a document but not common across all documents.

For a term \(t\) in document \(d\), the intuition is:

\begin{equation}
    \text{TF-IDF}(t,d) =
    \text{TF}(t,d) \times \text{IDF}(t).
\end{equation}

Term frequency measures how often a term appears in one document. Inverse document frequency reduces the weight of terms that appear in many documents and are therefore less discriminative. For harmful video detection, this is useful because generic words should have less influence than phrases or terms that are more specific to harmful or harmless contexts.

The system uses both unigrams and bigrams. Unigrams capture single words, while bigrams capture short two-word phrases. This helps the model represent phrases whose meaning depends on word combinations rather than isolated words.

\subsection{Logistic Regression for Binary Classification}

Logistic regression is a linear classification model that estimates the probability of a binary outcome. Given a feature vector \(x\), the model computes:

\begin{equation}
    P(y=1 \mid x) = \sigma(w^\top x + b),
\end{equation}

where \(w\) is the learned weight vector, \(b\) is the bias term, and \(\sigma\) is the sigmoid function:

\begin{equation}
    \sigma(z) = \frac{1}{1 + e^{-z}}.
\end{equation}

Logistic regression is a strong baseline for sparse text features because it is efficient, stable, and interpretable compared with heavier neural models. It is also appropriate for a local single-node workflow. In this project, probability calibration is applied so that predicted probabilities are more suitable for thresholding, risk-band mapping, and pseudo-label acceptance.

\subsection{Evaluation Metrics}

The project evaluates binary classification using multiple metrics rather than relying on accuracy alone. This is important because harmful-video datasets can be imbalanced, and a model may appear accurate while performing poorly on one class.

\begin{tabularx}{\textwidth}{|>{\raggedright\arraybackslash}p{3cm}|>{\raggedright\arraybackslash}X|}
\hline
\textbf{Metric} & \textbf{Meaning} \\
\hline
Precision & Among videos predicted as harmful, the proportion that are truly harmful. \\
\hline

\hline

\hline

\hline

\hline

\hline

Macro-F1 and harmful-class recall are emphasized in this project. Macro-F1 prevents the majority class from dominating the evaluation, while harmful-class recall measures how many harmful videos the model successfully detects. Precision is also important because false positives can incorrectly flag harmless videos, but the project prioritizes recall and macro-F1 for the core moderation-triage objective.

\subsection{Annotator Agreement}

Because the dataset includes labels from domain experts, GPT-4-Turbo, and crowdworkers, it is important to measure how much these sources agree. Two agreement measures are used: percent agreement and Cohen's kappa.

Percent agreement is the direct proportion of overlapping examples where two sources assign the same label:

\begin{equation}
    \text{Percent agreement} =
    \frac{\#\text{ matching labels}}{\#\text{ overlapping examples}}.
\end{equation}

However, percent agreement does not account for agreement that may happen by chance. Cohen's kappa adjusts for chance agreement:

\begin{equation}
    \kappa =
    \frac{p_o - p_e}{1 - p_e},
\end{equation}

where \(p_o\) is observed agreement and \(p_e\) is expected agreement by chance. A low kappa value can indicate that two sources have weak chance-adjusted consistency even when their raw percent agreement appears moderate.

In this project, agreement analysis is not only descriptive. It directly supports the weak-supervision design by showing that GPT and crowdworker labels should be treated as noisy sources rather than as equally reliable ground truth.
